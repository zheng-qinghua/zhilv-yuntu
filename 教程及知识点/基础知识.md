# 第一部分：Python 基础

## 1.1 `__init__.py` — Python 包机制

### 问题：为什么每个文件夹里都有一个空的 `__init__.py`？

Python 中，一个文件夹要想被当作"包"（package）来导入，必须包含一个 `__init__.py` 文件（即使内容为空）。

**没有 `__init__.py` 的后果**：

```python
# 如果 database/ 目录没有 __init__.py，这行会报错：
from database.connection import engine   # ModuleNotFoundError: No module named 'database'
```

**原理**：Python 解释器遇到 `from X import Y` 时，会去 `sys.path` 里搜索名为 `X` 的目录。如果 `X/` 目录下有 `__init__.py`，Python 就认为它是一个合法的包，允许从中导入模块。

**三种形态**：

```python
# 1. 空文件（本项目用的方式）
#    只标记"这是一个包"，不暴露任何内容

# 2. 在 __init__.py 中重导出（简化外部导入路径）
from database.chunk_repo import ChunkRepository  # __init__.py 内容
# 然后外部可以直接：from database import ChunkRepository
# 而不需要：from database.chunk_repo import ChunkRepository

# 3. 定义包级变量或初始化逻辑
print("database 包被加载了")   # import 时自动执行
```

## 1.2 Type Hints 类型标注

### 基础语法

Python 3.5+ 支持类型标注。标注不影响运行时行为，但能被 IDE 和类型检查器（mypy、pyright）用来提前发现错误。

```python
# 基本类型
name: str = "hello"           # 字符串
age: int = 18                 # 整数
price: float = 9.99           # 浮点数
is_active: bool = True        # 布尔值

# 容器类型（Python 3.9+）
names: list[str] = ["a", "b"]       # 字符串列表
scores: dict[str, int] = {"a": 1}   # 键为 str、值为 int 的字典
point: tuple[float, float] = (1.0, 2.0)

# 可选类型（Python 3.10+ 用 | 语法）
maybe_str: str | None = None         # 等价于 Optional[str]

# 函数签名
def greet(name: str) -> str:         # 参数类型 → 返回值类型
    return f"Hello {name}"
```

### `from __future__ import annotations` 的作用

```python
from __future__ import annotations
```

这一行让所有类型标注变成"惰性求值"——Python 不会在运行时检查标注，只当它们是注释。两个好处：

1. **避免循环导入**：A 模块标注引用了 B 的类型，B 又 import 了 A，普通情况会报错，加了这行就不会。
2. **向前兼容**：可以在低版本 Python 中用高版本语法（如 `str | None` 需要 3.10+，但加了 future annotations 后 3.7 也能运行）。

## 1.3 `if __name__ == "__main__"` — 模块入口

```python
if __name__ == "__main__":
    main()
```

**`__name__` 是什么**：每个 Python 文件都有一个内置变量 `__name__`，Python 根据"这个文件是怎么被执行的"来给它赋值：

| 执行方式 | `__name__` 的值 |
|----------|----------------|
| `python main.py`（直接运行） | `"__main__"` |
| `import main`（被别的文件导入） | `"main"`（模块名） |

**为什么要这样写**：防止被 import 时误执行代码。比如你写了一个工具库 `utils.py`，里面有个 `print("loaded")`。别人 `import utils` 时不应该触发打印，只有 `python utils.py` 时才触发。

```python
        return self.index is not None and len(self.id_map) > 0
```

## 1.5 `with` 上下文管理器

```python
# 等价写法对比

# 写法 A（手动管理，容易忘 close）
f = open("file.txt", "r")
try:
    content = f.read()
finally:
    f.close()

# 写法 B（with 语句，自动关闭）
with open("file.txt", "r") as f:
    content = f.read()
# 离开 with 块时，自动调用 f.close()
```

**原理**：`with` 调用对象的 `__enter__` 和 `__exit__` 方法。进入时执行 `__enter__`（获取资源），离开时无论是否异常都执行 `__exit__`（释放资源）。

项目中到处用了 `with SessionLocal() as session:`，离开时自动归还数据库连接到连接池。

# 第二部分：Pydantic 数据模型

## 2.1 什么是 Pydantic

Pydantic 是一个用类型标注做数据校验的库。你定义一个类，它自动获得：类型校验、序列化、JSON 互转、错误提示。

### 不用 Pydantic 的问题

```python
# 手写 dict 的问题
request = {
    "destination": "大理",
    "start_date": "2026-06-01",
    "budget": "五千",           # 拼写错误！应该是数字
    "travelers": -1,           # 负数！应该是 ≥1
}
# 直到 deep 进业务逻辑才发现问题，排查困难
```

### 用 Pydantic 之后

```python
from pydantic import BaseModel, Field

class TripRequest(BaseModel):
    destination: str
    budget: float = Field(..., ge=0)       # 必须 ≥0
    travelers: int = Field(default=1, ge=1) # 必须 ≥1

# 如果数据不合法：
try:
    req = TripRequest(destination="大理", budget=-100, travelers=0)
except Exception as e:
    print(e)
    # 输出清晰的 ValidationError:
    # budget: 必须 ≥0
    # travelers: 必须 ≥1
```

**数据一进入系统就被校验**，错误秒定位。

## 2.2 BaseModel 的核心能力

```python
from pydantic import BaseModel

class SpotItem(BaseModel):
    name: str
    estimated_cost: float = 0.0

# 1. 从 dict 创建（自动类型转换）
spot = SpotItem(name="大理古城", estimated_cost="0")    # "0" 自动转成 0.0

# 2. 导出为 dict
print(spot.model_dump())     # {'name': '大理古城', 'estimated_cost': 0.0}

# 3. 导出为 JSON
print(spot.model_dump_json()) # '{"name":"大理古城","estimated_cost":0.0}'

# 4. 从 JSON 创建
spot2 = SpotItem.model_validate_json('{"name":"洱海","estimated_cost":0.0}')
```

## 2.3 `Field()` 参数详解

```python
from pydantic import Field

# Field(...)  —— ...（Ellipsis）表示必填，没有默认值
name: str = Field(..., description="必填字段")

# Field(default=值)  —— 有默认值
pace: str = Field(default="适中")

# Field(default_factory=工厂函数)  —— 默认值是可变对象时必须用它
preferences: list[str] = Field(default_factory=list)
```

### `default_factory` 为什么不能直接用 `default=[]`

```python
# 错误写法（一个经典的 Python 坑）
class Bad(BaseModel):
    items: list[str] = []      # 所有实例共享同一个列表对象！

a = Bad()
b = Bad()
a.items.append("x")
print(b.items)  # ['x']  ← b 也被污染了！

# 正确写法
class Good(BaseModel):
    items: list[str] = Field(default_factory=list)  # 每次创建实例时新生成一个空列表
```

## 2.4 `model_validate()` 校验过程

```python
from pydantic import BaseModel

class DayPlan(BaseModel):

# model_validate(dict) 做的事：
# 1. 检查必填字段是否都有

# 3. 检查约束是否满足（ge=1 不能给 0）
# 4. 如果需要，自动类型转换（"3" → 3）
# 5. 返回一个 DayPlan 实例

```

## 2.5 Pydantic vs dataclass vs 手写 `__init__`

| 特性 | Pydantic | dataclass | 手写 `__init__` |
|------|----------|-----------|----------------|
| 类型校验 | 自动 | 需要 mypy | 无 |
| JSON 互转 | 内置 | 需要额外代码 | 需要额外代码 |
| 约束声明 (ge=1) | Field() | 需要额外库 | 手写 if |

**结论**：数据模型多、需要校验和 JSON 互转 → 选 Pydantic。

# 第三部分：SQLAlchemy + MySQL 数据库

## 3.1 MySQL 基础概念

### 数据库 vs 表 vs 行 vs 列

```
数据库 (Database) ── 像一个 Excel 文件
  └── 表 (Table) ── 像一个 Sheet 工作表
       ├── 列 (Column) ── 字段，如 id, title, text
       └── 行 (Row) ── 一条记录
```

### VARCHAR vs TEXT vs LONGTEXT

| 类型 | 最大长度 | 适用场景 |
|------|---------|---------|
| VARCHAR(N) | N 个字符（最大 65535 字节） | 短文本：ID、标题、文件名 |
| TEXT | 65535 字符 | 中等文本：正文、简介 |
| LONGTEXT | 4GB | 大文本：JSON 数组、向量 |

项目中 `embedding` 字段存的是 JSON 格式的浮点数组（如 `[0.0123, -0.0456, ...]`），1024 维约 8KB，用 TEXT 足够。

### 主键 (Primary Key)

主键是每行的唯一标识，不能重复、不能为空。项目中用 `source + MD5` 的组合作为主键。

### ON DUPLICATE KEY UPDATE

```sql
INSERT INTO guide_chunks (id, title, text) VALUES ('abc', '标题', '正文')
ON DUPLICATE KEY UPDATE
    title = VALUES(title),      -- 如果 id 冲突，更新 title
    text = VALUES(text);        -- 和 text
```

**意义**：重复执行 ingest 不会报错，会自动覆盖旧数据。这叫"幂等"——执行一次和执行一百次的效果相同。

---

## 3.2 SQLAlchemy 的定位

SQLAlchemy 不是数据库，是**操作数据库的 Python 工具包**。它提供两个层次：

- **Core 层**：连接池 + 原生 SQL（项目主要用这个）
- **ORM 层**：表 → 类，行 → 对象，SQL → 方法调用（项目只在建表时用）

```python
# Core 层（项目中用的方式）
session.execute(text("SELECT * FROM guide_chunks WHERE id = :id"), {"id": "abc"})

# ORM 层（项目建表时用）
class GuideChunk(Base):
    __tablename__ = "guide_chunks"
    id = Column(String(255), primary_key=True)
```

### 为什么项目主要用 Core 而非 ORM？

ORM 对于 CRUD 很方便，但复杂查询（如 `IN (?, ?, ?)` 动态参数）反而更绕。对于数据量少、只用简单 SQL 的场景，Core 层更容易控制和理解。

---

## 3.3 连接池原理

```python
from sqlalchemy import create_engine

engine = create_engine(
    DATABASE_URL,
    pool_size=5,          # 保持 5 个常驻连接
    max_overflow=10,      # 高峰时最多额外再建 10 个（共 15 个）
    pool_pre_ping=True,   # 取连接前发 SELECT 1，确认连接没断开
)
```

### 为什么需要连接池

TCP 连接建立成本很高（三次握手 + MySQL 认证），每次查询都新建连接会非常慢。连接池预先建好一批连接放在池里，需要时借一个，用完还回去。

```
┌─────────────┐      borrow       ┌──────────────┐
│   请求线程   │ ────────────────→ │  连接池 (5个) │ ──→ MySQL
│             │ ←──────────────── │              │
└─────────────┘      return       └──────────────┘
```

**各参数含义**：
- `pool_size=5`：连接池空闲时保持 5 个连接。全部借出后，第 6 个请求需要等待或创建溢出连接。
- `max_overflow=10`：高峰时池满之后，最多可以额外创建 10 个临时连接。高峰过后这些连接会关闭，恢复到 pool_size。
- `pool_pre_ping=True`：MySQL 默认 8 小时无活动会断开连接。取连接之前发一个 `SELECT 1` 确认连接还活着，死掉的自动重连。

## 3.4 Session（会话）的生命周期

```python
from database.connection import SessionLocal

# Session = 一次数据库会话 = 一个事务边界

with SessionLocal() as session:
    # __enter__: 从连接池获取一个连接，开启事务
    session.execute(text("SELECT ..."))
    session.execute(text("INSERT ..."))
    session.commit()     # 提交事务 —— 写入生效
    # __exit__:  自动归还连接到连接池
    #            如果中途抛异常没 commit，自动 rollback（回滚）
```

**SessionLocal 不是线程安全的**——每个线程/请求应该用独立的 session。

---

## 3.5 参数化查询（防 SQL 注入）

```python
# 危险写法——绝对不要用！
user_input = "'; DROP TABLE guide_chunks; --"
session.execute(text(f"SELECT * FROM guide_chunks WHERE id = '{user_input}'"))
# 拼接后: SELECT * FROM guide_chunks WHERE id = ''; DROP TABLE guide_chunks; --'
# 表被删了！！！

# 安全写法——参数化查询
user_input = "'; DROP TABLE guide_chunks; --"
session.execute(
    text("SELECT * FROM guide_chunks WHERE id = :id"),
    {"id": user_input}
)
# 数据库把 :id 当作纯字符串参数处理，特殊字符被转义，不会当成 SQL 执行
```

**本质**：参数化查询把"SQL 结构"和"数据值"分开传递。数据库先编译 SQL 模板，再把参数值填充进去，保证值永远不会被当作 SQL 代码解释。

## 3.6 ORM 建表 — `declarative_base`

```python
from sqlalchemy.orm import declarative_base
Base = declarative_base()

class GuideChunk(Base):

    id = Column(String(255), primary_key=True)    # 列定义
```

**`Base.metadata`** 是一个注册表，记录所有继承自 `Base` 的类对应的表结构。`create_all` 遍历它，生成 `CREATE TABLE IF NOT EXISTS ...`。

# 第四部分：Embedding（向量化）

## 4.1 什么是 Embedding

**一句话**：把文本变成一串数字（向量），使语义相似的文本在向量空间中距离近。

```
"大理古城"       →  [0.12, -0.45, 0.78, ..., 0.33]  (1024 个浮点数)
"南诏国都城遗址"  →  [0.11, -0.43, 0.79, ..., 0.31]  ← 向量很接近！
"今天天气不错"    →  [-0.67, 0.21, -0.15, ..., -0.02] ← 向量差很远！
```

### 为什么不能直接搜关键词

用户搜"慢节奏看日落"，但攻略里写的是"洱海生态廊道骑行，傍晚时分湖面金光非常出片"——完全没有"慢节奏"三个字。关键词匹配会漏掉这个好结果。

Embedding 模型理解语义，"慢节奏看日落" 和 "洱海傍晚出片" 在向量空间中距离很近，能被检索出来。

---

## 4.2 Embedding 模型原理（简述）

Embedding 模型（如 BAAI/bge-m3、text-embedding-3-small）是基于 Transformer 架构的神经网络，用数十亿条文本-文本对训练出来的。训练目标是：让相似文本的向量内积尽量大，不相似文本的向量内积尽量小。

调用 Embedding API 时，你的文本被送入这个预训练好的模型，模型输出中间层的激活值作为向量。你不需要训练模型，直接用现成的。

---

## 4.3 向量空间和维度

向量的每个维度编码了一种语义特征。比如第 0 维可能跟"是否与建筑有关"相关，第 1 维跟"是否与水有关"相关……（实际维度含义不可解释，是神经网络自动学的）。

- 常见维度：768（bert-base）、1024（BAAI/bge-m3）、1536（OpenAI text-embedding-3-small）
- 维度越高，表达能力越强，但存储和计算成本也越大

---

## 4.4 批量调用的意义

```python
# 差的做法：每条文本发一次 HTTP 请求
for text in texts:
    vector = api.embed(text)    # 50 条文本 = 50 次 HTTP 往返

# 好的做法：一次请求发 N 条
vectors = api.embed_batch(texts) # 50 条文本 = 1 次 HTTP 往返
```

HTTP 请求有固定的建立连接开销（几十毫秒），50 次 × 50ms = 2.5 秒光花在网络上了。批量接口把这部分开销从 N 次压缩到 1 次。

# 第五部分：FAISS 向量索引

## 5.1 FAISS 是什么

FAISS（Facebook AI Similarity Search）是 Meta 开源的高性能向量相似度搜索库。

- 做什么：在数百万个向量中快速找到与 query 向量最相似的 top_k 个
- 核心思想：把"逐一比较向量 × N"变成了高效的数据结构（索引）
- 类比：书本的目录就是正文的"索引"——不用翻全书就能找到某一章

---

## 5.2 `IndexFlatIP` — 暴力内积搜索

```python
import faiss

dimension = 1024                          # 向量维度
index = faiss.IndexFlatIP(dimension)      # 创建索引
index.add(vectors)                        # 把 N 个向量加入索引
distances, ids = index.search(query_vec, 5)  # 找最相似的 5 个
```

**"Flat"** = 暴力（不建索引结构，直接逐个比较）

**"IP"** = Inner Product（内积）

### 内积和余弦相似度

两个向量 A、B 的内积定义：

```
A · B = A[0]×B[0] + A[1]×B[1] + ... + A[D-1]×B[D-1]
```

余弦相似度定义：

```
cos(A, B) = (A · B) / (|A| × |B|)
```

如果 A 和 B 都是**单位向量**（长度=1，即 |A|=|B|=1），则：

```
cos(A, B) = A · B   （内积 = 余弦相似度）
```

所以流程是：

1. 把所有向量做 L2 归一化（变成单位向量）
2. 用 IndexFlatIP 查内积最大的 k 个
3. 返回的内积值就是余弦相似度

---

## 5.3 L2 归一化的数学含义

```python
import numpy as np

vectors = np.array([[3.0, 4.0], [1.0, 1.0]])   # 两个二维向量

# 计算每个向量的 L2 范数（模长）
norms = np.linalg.norm(vectors, axis=1, keepdims=True)
# [[5.0],    ← sqrt(3² + 4²) = 5
#  [1.414]]  ← sqrt(1² + 1²) = 1.414

# 归一化（每个向量除以自己的模长）
normalized = vectors / norms
# [[0.6, 0.8],      ← 这个向量的模长现在是 1
#  [0.707, 0.707]]  ← 这个也是 1
```

归一化后，所有向量的箭头都落在单位球面上，长度都是 1。此时两向量夹角越小（越相似），内积越大（最大为 1，最小为 -1）。

---

## 5.4 `keepdims=True` 的作用

```python
a = np.array([[1, 2], [3, 4]])  # shape (2, 2)

# 不保持维度
np.linalg.norm(a, axis=1)        # [2.236, 5.0], shape (2,)

# 保持维度
np.linalg.norm(a, axis=1, keepdims=True)  # [[2.236], [5.0]], shape (2, 1)
```

如果维度不匹配，NumPy 的广播机制行为不同。`(2, 2) / (2,)` 和 `(2, 2) / (2, 1)` 结果一样但容易出错，统一用 `keepdims=True` 更安全。

---

## 5.5 Flat vs 近似索引

| 索引类型 | 原理 | 速度 | 精度 | 适用规模 |
|---------|------|------|------|---------|
| IndexFlatIP | 暴力比较 | O(N) | 100% | < 10 万 |
| IndexIVFFlat | 先聚类，只搜最近几个簇 | O(log N) | 95%+ | 10 万 ~ 1000 万 |
| IndexHNSWFlat | 图结构，多层跳跃搜索 | O(log N) | 98%+ | 1 万 ~ 100 万 |

项目中文档块最多几百条，Flat 完全够用，而且能保证 100% 精确结果。

# 第六部分：RAG 完整流程

## 6.1 什么是 RAG

**RAG** = Retrieval Augmented Generation（检索增强生成）

传统 LLM 的问题：
- 知识截止于训练日期，不知道最新信息
- 可能"幻觉"（编造不存在的事实）
- 无法访问私有/本地文档

RAG 解决方案：**先检索相关文档，再把文档塞给 LLM 做参考，最后生成回答。**

```
用户问题 ──→ [检索] ──→ 找到相关文档片段
                              │
                              ▼
            LLM + 用户问题 + 文档片段 ──→ 有据可查的回答
```

---

## 6.2 本项目 RAG 的完整五步

```
Step 1: Query Rewrite（查询改写）
  用户: "大理慢节奏看日落拍照"
    │
    ▼ LLM 改写
  "大理 洱海 双廊 日落 拍照 轻松 攻略 推荐"

Step 2: Embedding（向量化）
  "大理 洱海 双廊 日落 拍照 轻松 攻略 推荐" → [0.01, -0.05, 0.12, ...]

Step 3: FAISS 粗排（召回 stage）
  Query 向量 在 IndexFlatIP 中搜索 top_k×2 = 10 个候选 chunk
  召回率优先——宁可多召回一些，下一步再精排

Step 4: 精排（Rerank stage）
  对 10 个候选用规则打分（标题命中 +3、正文命中 +1、噪声 -8...）
  排序后取 top_k = 5 个
  精确率优先——确保给 LLM 看的是最相关的

Step 5: 回 MySQL 取完整文本
  FAISS 只返回 chunk_id 和相似度分数
  去 MySQL 用 SELECT ... WHERE id IN (...) 取完整 title + text
```

---

## 6.3 粗排 vs 精排

| | 粗排（Recall） | 精排（Rerank） |
|------|------|------|
| 目标 | 尽量多召回，不怕多 | 精确排序，只要最好的 |
| 数据量 | 数百~数千个候选 | 几十个候选 |
| 速度 | 毫秒级（FAISS） | 毫秒级（规则）/百毫秒级（LLM） |
| 原则 | 宁可错杀一千，不可放过一个 | 宁缺毋滥 |

# 第七部分：LangChain

## 7.1 LangChain 是什么

LangChain 是一个 LLM 应用开发框架。它的核心价值是把 LLM 调用从"发 HTTP 请求"抽象成"调函数"：

```python
# 不用 LangChain（自己拼 HTTP 请求）
import httpx
resp = httpx.post("https://api.deepseek.com/v1/chat/completions", json={
    "model": "deepseek-v4-flash",
    "messages": [
        {"role": "system", "content": "你是助手"},
        {"role": "user", "content": "你好"}
    ]
})
text = resp.json()["choices"][0]["message"]["content"]

# 用 LangChain（封装了 HTTP 细节）
from langchain_openai import ChatOpenAI
llm = ChatOpenAI(model="deepseek-v4-flash", base_url="...")
response = llm.invoke([
    ("system", "你是助手"),
    ("human", "你好"),
])
text = response.content
```

LangChain 还提供 Chain（链式调用）、Memory（对话记忆）、Agent（工具调用）等高级抽象——但本项目只用最基本的 ChatModel 封装。

---

## 7.2 `ChatOpenAI` 为什么能调用 DeepSeek

DeepSeek 的 API **完全兼容 OpenAI 的 Chat Completions 格式**。请求体结构一样：

```json
{
  "model": "deepseek-v4-flash",
  "messages": [
    {"role": "system", "content": "..."},
    {"role": "user", "content": "..."}
  ],
  "temperature": 0.3
}
```

所以 `ChatOpenAI` 只需要改 `base_url` 指向 DeepSeek 的地址就行，其他代码不用变。

---

## 7.3 `invoke()` 方法

```python
response = llm.invoke([
    ("system", "你是一个助手"),
    ("human", "你好"),
])
```

`invoke()` 是同步调用——发送请求后**阻塞等待**，直到 LLM 返回完整结果。返回的是 `AIMessage` 对象：

```python
print(response.content)           # "你好！有什么可以帮你的？"
print(response.response_metadata) # {"token_usage": {"prompt_tokens": 10, "completion_tokens": 5}, ...}
```

---

## 7.4 System Prompt vs Human Prompt

| 类型 | 角色 | 作用 | 示例 |
|------|------|------|------|
| System | 设定 AI 的行为规则 | 限制输出格式、定义身份 | "你只输出 JSON，不要解释" |
| Human | 具体任务输入 | 提供上下文和指令 | "目的地：大理\n天数：3" |

---

## 7.5 `temperature` 参数

```python
llm = ChatOpenAI(model="deepseek-v4-flash", temperature=0.3)
```

`temperature` 控制输出的**随机性**，范围 0~2：

- `temperature ≈ 0`：每次输出几乎相同（确定性强），适合格式化输出（JSON）
- `temperature ≈ 0.3-0.5`：有一定变化但可控，适合旅行规划（有小变化但不过分）
- `temperature ≈ 1.0+`：变化很大，适合创意写作（写诗、故事）

项目中 Query Rewrite 用 0.2（要稳定的关键词），行程生成用 0.3（稍多一点变化但不过分）。

## 7.6 Token 用量提取

```python
response = llm.invoke([...])

# response.response_metadata 里有 token 统计
metadata = response.response_metadata
# {"token_usage": {"prompt_tokens": 350, "completion_tokens": 200}, ...}

token_usage = metadata.get("token_usage", {})
print(token_usage["prompt_tokens"])     # 输入花了多少 token
print(token_usage["completion_tokens"]) # 输出花了多少 token
```

**Token** 是 LLM 计费的基本单位。一个中文字 ≈ 1-2 个 token。监控 token 用量可以帮助控制成本。

# 第八部分：设计模式

## 8.1 单例模式（Singleton）

**目的**：全局只有一个实例。

**Python 模块级单例**：

```python
# config.py
LLM_API_KEY = os.getenv("LLM_API_KEY", "")

# 其他文件
from config import LLM_API_KEY   # LLM_API_KEY 整个进程只被创建一次
```

Python 的 import 机制天然实现了单例——第一次 `import config` 时执行模块代码，后续 `import config` 直接返回缓存的模块对象。不需要显式写 `class Singleton`。

`database/connection.py` 同理：`engine` 和 `SessionLocal` 是模块级变量，全局共享。

---

## 8.2 DAO 模式（Data Access Object）

**目的**：把数据库操作封装在一个类里，业务逻辑不接触 SQL。

```python
# 不用 DAO（SQL 散落各处）
# trip_service.py 里：
session.execute(text("INSERT INTO guide_chunks ..."))
# agent.py 里：
session.execute(text("SELECT * FROM guide_chunks WHERE ..."))
# 改表结构要改 10 个文件！

# 用 DAO（SQL 集中管理）
class ChunkRepository:
    def insert_chunk(self, ...): ...
    def get_chunks_by_ids(self, ...): ...
    def count_chunks(self, ...): ...

# 业务代码只调方法
repo.insert_chunk(...)
chunks = repo.get_chunks_by_ids(ids)
```

改一张表的字段，只改 ChunkRepository 一个类。

---

## 8.3 门面模式（Facade）

**目的**：对外暴露简单接口，隐藏内部复杂性。

```python
class Retriever:
    def retrieve(self, query: str, top_k: int = 5) -> list[str]:
        """外部只需要知道这一个方法"""
        # 内部做了：
        # 1. 调 embedding API
        # 2. 搜 FAISS
        # 3. 回 MySQL 取文本
        # 4. 规则打分精排
        # 5. 格式化返回
        ...
```

调用方只需要 `retriever.retrieve("大理攻略")` 就能拿到结果，完全不知道内部怎么做的。改了内部实现也不会影响调用方。

---

## 8.4 策略模式（Strategy）

**目的**：把可替换的算法封装成独立对象，运行时可以切换。

```python
class EmbeddingService:
    """当前用 HTTP API 做 embedding"""
    def embed_batch(self, texts): ...

# 将来可以无痛替换：
class LocalEmbeddingService:
    """用本地 ONNX 模型做 embedding，不调 API，零成本"""
    def embed_batch(self, texts): ...

# 上层代码 Retriever 不需要改动——它只依赖"能 embed_batch 的东西"
```

---

## 8.5 依赖注入（Dependency Injection）

**目的**：对象不自己创建依赖，而是从外部接收。

```python
# 不用依赖注入（硬编码依赖，紧耦合）
class Retriever:
    def __init__(self):
        self.vector_index = VectorIndex()        # 自己 new，写死了
        self.embed_service = EmbeddingService()  # 没法替换

# 用依赖注入（松耦合）
class Retriever:
    def __init__(self, vector_index, embedding_service, chunk_repo):
        self.vector_index = vector_index          # 从外面传进来
        self.embedding_service = embedding_service # 可以传 mock
        self.chunk_repo = chunk_repo
```

**好处**：
- 测试时传 mock 对象，不调真实 API
- 换实现时只改组装代码（main.py），不改类内部
- 依赖关系一目了然

# 第十部分：其他技术点

## 10.1 MD5 哈希做幂等 ID

```python
from hashlib import md5

text = "dali_guide.md|洱海生态廊道|洱海西岸的慢行系统..."
digest = md5(text.encode("utf-8")).hexdigest()
# digest = "a1b2c3d4e5f6..."  (32 字符的十六进制字符串)
```

**MD5 的特点**：
- 相同输入 → 相同输出（确定性）
- 不同输入 → 极大概率不同输出（碰撞概率极低）
- 任意长度输入 → 固定 128 位（32 字符 hex）输出

**为什么用 MD5 而不是随机 UUID**：相同内容的 chunk 在多次 ingest 后生成相同的 ID，MySQL 的 `ON DUPLICATE KEY UPDATE` 自动覆盖，而不是产生重复记录。如果用随机 UUID，每次都会新增。
