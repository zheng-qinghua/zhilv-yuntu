# 智旅云图 — 后端 Agent + RAG 核心从零搭建教程

## 项目目标

在命令行输入一个地点（如"大理"），Agent 通过 RAG 检索本地旅游攻略文档，调用大模型生成一份结构化的旅行规划。

**技术栈**: Python + LangChain + DeepSeek API + MySQL + FAISS

---

## 第 0 步：环境准备

### 0.1 安装 MySQL

去 https://dev.mysql.com/downloads/mysql/ 下载 MySQL 8.0+ 安装包，安装时记住你设置的 **root 密码**。

安装完成后，打开命令行，用 root 用户登录 MySQL，创建一个专用数据库：

```sql
mysql -u root -p

-- 输入你安装时设置的 root 密码

CREATE DATABASE travel_agent CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;

-- 创建一个专用用户（可选，也可以直接用 root）
CREATE USER 'travel'@'localhost' IDENTIFIED BY 'your_password_here';
GRANT ALL PRIVILEGES ON travel_agent.* TO 'travel'@'localhost';
FLUSH PRIVILEGES;

EXIT;
```

记下数据库连接信息，后面要写入 `.env`：

| 配置项 | 值 |
|--------|-----|
| 主机 | localhost |
| 端口 | 3306 |
| 数据库名 | travel_agent |
| 用户名 | travel（或 root） |
| 密码 | 你自己设的 |

### 0.2 申请 DeepSeek API Key

1. 打开 https://platform.deepseek.com
2. 注册/登录，进入"API Keys"页面
3. 点击"创建 API Key"，复制保存

### 0.3 申请 Embedding API（二选一）

DeepSeek 目前不提供 Embedding 模型，你需要单独申请。推荐以下免费/便宜的方案：

**方案 A（推荐国内用户）**：硅基流动 SiliconFlow
- 地址: https://siliconflow.cn
- 注册后在"API 密钥"页面创建 Key
- 免费额度足够开发和测试用
- 模型名: `BAAI/bge-m3`（中文效果最好的免费 Embedding 模型之一）

**方案 B**：OpenAI 官方
- 地址: https://platform.openai.com
- 模型名: `text-embedding-3-small`
- 按量计费，每 1000 token 约 $0.00002

# ============================================================

import os

## 第五步：`database/connection.py` MySQL 连接池

**设计思路**：使用 **单例模式** 管理 MySQL 连接池。`create_engine` 本身就是连接池——它内部维护一组 TCP 连接，需要时取一个，用完归还。全局只创建一个 engine 实例，所有数据库操作共享。

```python
# ============================================================
# database/connection.py — MySQL 连接池管理
# ============================================================

# 1. create_engine：SQLAlchemy 核心工厂函数
#    参数说明：
#      - DATABASE_URL: 连接字符串
#      - pool_size=5: 连接池最多保持 5 个连接
#      - max_overflow=10: 高峰时可额外再创建 10 个（共 15 个）
#      - pool_pre_ping=True: 每次取出连接前先发一个 SELECT 1 探测是否存活
#                            如果连接已被 MySQL 服务端断开，自动重连
#      - echo=False: 不打印 SQL 日志（调试时可改为 True）
from sqlalchemy import create_engine

# 2. sessionmaker：创建 Session 工厂
#    每个 Session 代表一次数据库会话（事务边界）
#    autocommit=False: 需要手动 commit
#    autoflush=False: 不自动刷新——改为手动控制
from sqlalchemy.orm import sessionmaker

# 3. declarative_base：声明式 ORM 基类
#    虽然本项目用原生 SQL 而非 ORM，但建表脚本需要它
from sqlalchemy.orm import declarative_base

from config import DATABASE_URL


# ---- 创建引擎（连接池） ----
engine = create_engine(
    DATABASE_URL,
    pool_size=5,            # 常规连接数
    max_overflow=10,        # 溢出连接数
    pool_pre_ping=True,     # 连接健康检查
    echo=False,             # 不打印 SQL
)

# ---- 创建 Session 工厂 ----
# SessionLocal 是一个可调用对象，每次调用创建一个新的数据库会话
# 使用方式:
#   with SessionLocal() as session:
#       session.execute(...)
#       session.commit()
SessionLocal = sessionmaker(autocommit=False, autoflush=False, bind=engine)

# ---- ORM 基类（建表用） ----
Base = declarative_base()
```

## 第六步：`database/init_db.py` 建表脚本

**设计思路**：建表逻辑独立——表结构变更只需改这一个文件。使用 SQLAlchemy ORM 的 `Base.metadata.create_all` 自动建表，比手写 `CREATE TABLE` SQL 更安全（自动处理 IF NOT EXISTS）。

```python
# ============================================================
# database/init_db.py — 数据库初始化
# ============================================================

# Column: 列定义
# Integer, String, Text, Float, DateTime: 列类型
from sqlalchemy import Column, Integer, String, Text, Float, DateTime

# func: SQL 函数（如 func.now() → NOW()）
from sqlalchemy.sql import func

from database.connection import Base, engine


class GuideChunk(Base):
    """
    攻略文档块表。
    每一行 = Markdown 攻略按标题切分后的一个文本块 + 它的向量表示。

    列说明:
      id          主键，用 source + MD5 生成，如 "dali_guide.md_a1b2c3d4"
      title       该文本块所属的标题，如 "洱海生态廊道"
      text        文本块的正文（去掉了标题行）
      source      来源文件名，如 "dali_guide.md"
      embedding   文本转向量后的浮点数组，存为 JSON 字符串
                  例如 "[0.0123, -0.0456, 0.0789, ...]"
      created_at  入库时间，自动填充
    """
    __tablename__ = "guide_chunks"

    # primary_key=True: 主键
    id = Column(String(255), primary_key=True)

    # nullable=False: 不允许为空
    title = Column(String(500), nullable=False)

    # Text: 长文本类型（无长度限制，区别于 VARCHAR）
    text = Column(Text, nullable=False)

    source = Column(String(255), nullable=False)

    # LONGTEXT: MySQL 的长文本，存 JSON 格式的向量（1024-4096 个浮点数）
    # 一个 1024 维向量序列化成 JSON 大约 8KB
    embedding = Column(Text, nullable=True)

    # server_default=func.now(): 由 MySQL 服务端在 INSERT 时自动填当前时间
    created_at = Column(DateTime, server_default=func.now())


def init_database():
    """
    创建所有表（如果表已存在则跳过）。
    程序启动时调用一次即可。
    """
    Base.metadata.create_all(bind=engine)
    print("[database] 数据库表创建完成")


if __name__ == "__main__":
    # 允许单独运行此文件来建表: python -m database.init_db
    init_database()
```

## 第七步：`database/chunk_repo.py` 数据访问层（DAO 模式）

**设计思路**：使用 **DAO（Data Access Object）模式**——数据库操作全部封装在 `ChunkRepository` 类中。上层代码不直接写 SQL，而是调用对象方法。接口清晰、方便单元测试时 Mock。

```python
# ============================================================
# database/chunk_repo.py — 攻略文本块的增删改查
# ============================================================

import json
from typing import Optional

# text(): 构造原生 SQL 语句
from sqlalchemy import text

# 连接池
from database.connection import SessionLocal


class ChunkRepository:
    """
    攻略文本块的数据访问对象。

    设计模式: DAO (Data Access Object)
    职责: 封装所有对 guide_chunks 表的操作
    优势:
      - 上层代码不接触 SQL，修改表结构只改这一个类
      - 可以 mock 掉做单元测试
    """

    def insert_chunk(
        self,
        chunk_id: str,
        title: str,
        text_content: str,
        source: str,
        embedding: list[float] | None = None
    ) -> None:
        """
        插入一条文本块。

        参数:
          chunk_id:   唯一 ID，由 source + MD5 生成
          title:      标题，如 "洱海生态廊道"
          text_content: 正文
          source:     来源文件名，如 "dali_guide.md"
          embedding:  向量（浮点列表），可选，入库时一般有值
        """
        # json.dumps() 把 Python list 转成 JSON 字符串存入 MySQL
        embedding_json = json.dumps(embedding) if embedding else None

        # with SessionLocal() as session:
        #   __enter__ 从连接池获取一个连接
        #   __exit__  自动归还连接（或回滚）
        with SessionLocal() as session:
            # session.execute(): 执行原生 SQL
            # text() 把字符串包装成 SQLAlchemy 可执行的 SQL 对象
            # :id, :title 是命名参数，用后面的 dict 提供值
            # 参数化查询防止 SQL 注入
            session.execute(
                text("""
                    INSERT INTO guide_chunks (id, title, text, source, embedding)
                    VALUES (:id, :title, :text, :source, :embedding)
                    ON DUPLICATE KEY UPDATE
                        title = VALUES(title),
                        text = VALUES(text),
                        embedding = VALUES(embedding)
                """),
                {
                    "id": chunk_id,
                    "title": title,
                    "text": text_content,
                    "source": source,
                    "embedding": embedding_json,
                }
            )
            # commit() 提交事务——写入真正生效
            session.commit()

    def get_all_chunks(self) -> list[dict]:
        """
        查询所有文本块（不含 embedding，加载索引时才查 embedding）。

        返回示例:
          [
            {"id": "dali_guide.md_abc123", "title": "洱海", "text": "...", "source": "dali_guide.md"},
            ...
          ]
        """
        with SessionLocal() as session:
            # session.execute() 返回 Result 对象
            result = session.execute(
                text("SELECT id, title, text, source FROM guide_chunks")
            )
            # result.mappings() 把每一行转成 dict-like 对象
            # dict(row) 得到真正的 Python dict
            return [dict(row) for row in result.mappings()]

    def get_all_embeddings(self) -> list[dict]:
        """
        查询所有文本块的 ID 和向量（建 FAISS 索引时用）。
        只查 id + embedding，减少数据传输量。

        返回示例:
          [
            {"id": "dali_guide.md_abc123", "embedding": [0.0123, -0.0456, ...]},
            ...
          ]
        """
        with SessionLocal() as session:
            result = session.execute(
                text("SELECT id, embedding FROM guide_chunks WHERE embedding IS NOT NULL")
            )
            rows = []
            for row in result.mappings():
                row_dict = dict(row)
                # json.loads() 把 JSON 字符串还原成 Python list
                if row_dict["embedding"]:
                    row_dict["embedding"] = json.loads(row_dict["embedding"])
                rows.append(row_dict)
            return rows

    def get_chunks_by_ids(self, chunk_ids: list[str]) -> list[dict]:
        """
        根据 ID 列表查询文本块（检索后回表取完整数据用）。

        数据流: FAISS 搜索返回 ID 列表 → 用这个方法取完整文本。
        """
        if not chunk_ids:
            return []
        with SessionLocal() as session:
            # 用 IN 子句批量查询
            # 注意：参数化查询中 IN 的占位符需要动态拼接
            # :id_0, :id_1, :id_2 ...
            placeholders = ", ".join(f":id_{i}" for i in range(len(chunk_ids)))
            params = {f"id_{i}": cid for i, cid in enumerate(chunk_ids)}
            result = session.execute(
                text(f"SELECT id, title, text, source FROM guide_chunks WHERE id IN ({placeholders})"),
                params
            )
            return [dict(row) for row in result.mappings()]

    def count_chunks(self) -> int:
        """查询文本块总数（检查是否已入库用）。"""
        with SessionLocal() as session:
            result = session.execute(text("SELECT COUNT(*) AS cnt FROM guide_chunks"))
            # scalar(): 返回第一行第一列的值
            return result.scalar() or 0

    def delete_all(self) -> None:
        """清空表（重新 ingest 前用）。"""
        with SessionLocal() as session:
            session.execute(text("DELETE FROM guide_chunks"))
            session.commit()

## 第九步：`rag/chunker.py` Markdown 文本切分

**设计思路**：把 Markdown 文件按 `##` / `###` 标题切分成独立文本块（chunk）。每个标题段落作为一个检索单元——这样用户搜"洱海骑行"时能精确命中"洱海生态廊道"段落，而不是整篇大理攻略。

```python
# ============================================================
# rag/chunker.py — Markdown → 文本块列表
# ============================================================

# hashlib.md5(): 生成内容的 MD5 摘要，用作 chunk 的唯一 ID
# 相同内容 → 相同 ID，实现幂等（重复 ingest 不会产生重复记录）
from hashlib import md5

from config import DATA_DIR


def split_markdown_to_chunks(markdown_text: str, source_name: str) -> list[dict[str, str]]:
    """
    把一篇 Markdown 攻略按 ## / ### 标题切分成多个 chunk。

    算法逻辑（逐行状态机）:
      1. 遍历每一行
      2. 遇到 ## 或 ### 开头 → 保存上一个 chunk，开始新 chunk
      3. 普通文本行 → 追加到当前 chunk
      4. 文件末尾 → 保存最后一个 chunk

    参数:
      markdown_text: Markdown 全文
      source_name: 文件名，如 "dali_guide.md"

    返回:
      [{"title": "洱海生态廊道", "text": "...", "source": "dali_guide.md"}, ...]
    """
    chunks: list[dict[str, str]] = []

    # 第一个标题前的文本归入 "文档开头"（后续检索时会降权，避免噪声）
    current_title = "文档开头"

    # 当前标题下的所有文本行
    current_lines: list[str] = []

    for line in markdown_text.splitlines():
        # strip(): 去掉首尾空格和换行符
        stripped = line.strip()

        # startswith("## "): 以 "## " 开头，即二级标题
        # 注意：不匹配 "### "，因为后面单独判断
        if stripped.startswith("## ") or stripped.startswith("### "):
            # 遇到新标题 → 保存 accumulating 的文本作为一个 chunk
            if current_lines:
                chunks.append({
                    "title": current_title,
                    # "\n".join() 把多行合成一个字符串
                    "text": "\n".join(current_lines).strip(),
                    "source": source_name,
                })
                current_lines = []

            # lstrip("#"): 去掉行首所有 # 号
            # strip(): 再去掉首尾空格
            current_title = stripped.lstrip("#").strip()

        elif stripped:
            # 非空行 → 追加到当前 chunk
            current_lines.append(stripped)
        # 空行 → 跳过

    # 循环结束后，保存最后一个 chunk
    if current_lines:
        chunks.append({
            "title": current_title,
            "text": "\n".join(current_lines).strip(),
            "source": source_name,
        })

    return chunks


def build_chunk_id(source: str, title: str, text: str) -> str:
    """
    生成稳定的 chunk ID。

    用 MD5 而非随机数：相同内容多次 ingest 生成相同 ID，
    MySQL INSERT ... ON DUPLICATE KEY UPDATE 自动覆盖而不是重复插入。

    参数:
      source: 文件名，如 "dali_guide.md"
      title:  标题，如 "洱海生态廊道"
      text:   正文

    返回:
      "dali_guide.md_a1b2c3d4e5f6..." （文件名 + 32 位 MD5 十六进制串）
    """
    # encode("utf-8"): 字符串 → 字节序列（MD5 只接受字节）
    # hexdigest(): 128 位摘要 → 32 字符的十六进制字符串
    digest = md5(f"{source}|{title}|{text}".encode("utf-8")).hexdigest()
    return f"{source}_{digest}"


def load_all_chunks() -> list[dict[str, str]]:
    """
    遍历 data/ 目录下所有 .md 文件，全部切分成 chunk。

    返回:
      [
        {"id": "dali_guide.md_abc123", "title": "洱海生态廊道", "text": "...", "source": "dali_guide.md"},
        {"id": "dali_guide.md_def456", "title": "喜洲古镇", "text": "...", "source": "dali_guide.md"},
        ...
      ]
    """
    all_chunks: list[dict[str, str]] = []

    # DATA_DIR.glob("*.md*"): 匹配 data/ 下所有 .md 和 .mdx 文件
    # sorted(): 按文件名排序，保证处理顺序一致
    for guide_file in sorted(DATA_DIR.glob("*.md*")):
        # read_text(encoding="utf-8"): Path 对象的便捷读文件方法
        # 等价于 open(file, "r", encoding="utf-8").read()
        text = guide_file.read_text(encoding="utf-8")

        for chunk in split_markdown_to_chunks(text, guide_file.name):
            all_chunks.append({
                "id": build_chunk_id(chunk["source"], chunk["title"], chunk["text"]),
                "title": chunk["title"],
                "text": chunk["text"],
                "source": chunk["source"],
            })

    return all_chunks
```

## 第十步：`rag/embedding.py` Embedding 服务

**设计思路**：把文本转成向量。封装 HTTP 调用细节（URL 拼接、Header、错误处理），上层代码只需要 `embed(["文本1", "文本2"])` → `[[0.01, -0.02...], [...]]`。

```python
# ============================================================
# rag/embedding.py — 文本 → 向量的转换服务
# ============================================================

# httpx: 现代化 HTTP 客户端，比 requests 更简洁
# Client(timeout=...): 创建带超时的连接池
import httpx

from config import (
    EMBEDDING_API_KEY,
    EMBEDDING_BASE_URL,
    EMBEDDING_BATCH_SIZE,
    EMBEDDING_MODEL,
)


class EmbeddingService:
    """
    Embedding 服务类。

    设计模式: 策略模式
    把"如何向量化"封装在一个对象中，方便替换实现
    （比如将来可以换成本地模型而不改上层代码）。

    使用示例:
      service = EmbeddingService()
      vectors = service.embed(["大理古城是...", "洱海位于..."])
      # vectors = [[0.01, -0.02, ...], [0.03, 0.01, ...]]
    """

    def __init__(self):
        """初始化 Embedding 服务的配置。"""
        # API Key —— 如果没有配置，后续调用会返回 None
        self.api_key = EMBEDDING_API_KEY

        # 模型名，如 "BAAI/bge-m3"
        self.model = EMBEDDING_MODEL

        # 基础 URL，去掉末尾的 / 再拼 /embeddings
        # 例如 "https://api.siliconflow.cn/v1" → "https://api.siliconflow.cn/v1/embeddings"
        self.base_url = (EMBEDDING_BASE_URL or "").rstrip("/")

        # 批次大小：每批最多发多少条文本
        self.batch_size = EMBEDDING_BATCH_SIZE

    def embed_single(self, text: str) -> list[float] | None:
        """
        把单个文本转换成向量。

        参数:
          text: 任意字符串，如 "大理古城是南诏国的都城遗址"

        返回:
          浮点列表，如 [0.0123, -0.0456, 0.0789, ...]（通常 1024 维）
          失败时返回 None
        """
        results = self.embed_batch([text])
        if results and len(results) > 0:
            return results[0]
        return None

    def embed_batch(self, texts: list[str]) -> list[list[float]] | None:
        """
        批量把文本转成向量。

        为什么用批量接口：
          单条调用一次 API 的 HTTP 开销很大。批量接口可以把 N 条文本
          打包成一个请求，一次返回 N 个向量，大幅减少网络往返。

        参数:
          texts: 文本列表

        返回:
          向量列表，失败时返回 None
        """
        if not self.api_key:
            print("[embedding] 未配置 EMBEDDING_API_KEY")
            return None

        # 构造请求 URL
        url = f"{self.base_url}/embeddings"

        # HTTP 请求体
        payload = {
            "model": self.model,
            "input": texts,          # API 接受 list[str] 批量输入
        }

        # HTTP 请求头
        headers = {
            "Authorization": f"Bearer {self.api_key}",
            "Content-Type": "application/json",
        }

        try:
            # httpx.Client 作为上下文管理器，自动关闭连接
            with httpx.Client(timeout=60) as client:
                resp = client.post(url, json=payload, headers=headers)

            if resp.status_code != 200:
                print(f"[embedding] API 返回错误 {resp.status_code}: {resp.text[:200]}")
                return None

            # resp.json() 解析 JSON 响应体
            data = resp.json()

            # OpenAI 兼容格式: {"data": [{"embedding": [...], "index": 0}, ...]}
            # 按 index 排序（并发返回时顺序可能打乱）
            items = sorted(data.get("data", []), key=lambda x: x.get("index", 0))

            # 提取每个 embedding
            vectors = [item["embedding"] for item in items if "embedding" in item]

            if len(vectors) != len(texts):
                print(f"[embedding] 返回数量不匹配: 期望 {len(texts)}, 实际 {len(vectors)}")
                return None

            print(f"[embedding] 成功向量化 {len(vectors)} 条文本")
            return vectors

        except Exception as exc:
            print(f"[embedding] 请求失败: {type(exc).__name__}: {exc}")
            return None
```

## 第十一步：`rag/vector_index.py` FAISS 向量索引

**设计思路**：FAISS 是 Facebook 开源的高性能向量相似度搜索库。我们用它在内存中为所有文本块的向量建索引，查询时输入 query 向量，返回最相似的 top_k 个 chunk ID。

**核心概念**：
- `IndexFlatIP`: 暴力内积搜索——query 向量与索引中每个向量算内积（等价于 cosine 相似度），取最大的 k 个
- ID Map: 把 FAISS 内部的整数序号映射回我们自己的 chunk ID

```python
# ============================================================
# rag/vector_index.py — FAISS 向量索引管理
# ============================================================

import json
import pickle

# numpy: 科学计算库，FAISS 的所有向量数据都用 np.float32 数组
# np.array([[1.0, 2.0], [3.0, 4.0]], dtype=np.float32)
import numpy as np

# faiss: 向量相似度搜索库
# IndexFlatIP: 内积搜索索引（Inner Product）
#   对于已归一化的向量，内积 = cosine 相似度
import faiss

from config import FAISS_INDEX_PATH, FAISS_ID_MAP_PATH


class VectorIndex:
    """
    FAISS 向量索引管理类。

    设计模式: 外观模式（Facade）
    封装 FAISS、numpy 等底层操作，对外提供简单的 build / search / save / load 接口。

    数据流:
      build: [chunk_id, embedding] → FAISS 索引 + ID 映射表
      search: query_embedding → top_k 个 chunk_id
    """

    def __init__(self):
        # index: FAISS 索引对象，初始为 None
        self.index: faiss.IndexFlatIP | None = None

        # id_map: 序号 → chunk_id 的映射
        # 例如 {0: "dali_guide.md_abc123", 1: "dali_guide.md_def456"}
        self.id_map: dict[int, str] = {}

        # 向量的维度（所有向量维度必须一致）
        self.dimension: int = 0

    def build(
        self,
        chunk_ids: list[str],
        embeddings: list[list[float]]
    ) -> None:
        """
        从 chunk ID 和向量数组构建 FAISS 索引。

        参数:
          chunk_ids: ["dali_guide.md_abc", "dali_guide.md_def", ...]
          embeddings: [[0.01, -0.02, ...], [0.03, 0.01, ...], ...]

        过程:
          1. 把每个向量做 L2 归一化（使内积等价于 cosine 相似度）
          2. 创建 IndexFlatIP（内积搜索索引）
          3. 把所有归一化后的向量加入索引
          4. 建立序号 → chunk_id 映射
        """
        if not embeddings:
            print("[vector_index] 没有向量数据，跳过索引构建")
            return

        # 转成 np.float32 矩阵，形状 (N, D)
        # N = chunk 数量, D = 向量维度
        vectors = np.array(embeddings, dtype=np.float32)
        self.dimension = vectors.shape[1]

        # ---- L2 归一化 ----
        # 把每个向量除以其 L2 范数（模长），使所有向量落在单位球面上
        # 归一化后，两个向量的内积 = cos(夹角) = 余弦相似度
        # np.linalg.norm(vectors, axis=1, keepdims=True):
        #   计算每个向量的 L2 范数，keepdims=True 保持二维形状便于广播除法
        norms = np.linalg.norm(vectors, axis=1, keepdims=True)
        # 避免除零
        norms = np.where(norms == 0, 1.0, norms)
        vectors = vectors / norms

        # ---- 创建 FAISS 索引 ----
        # IndexFlatIP: 暴力搜索（遍历所有向量算内积）
        # 优点: 精确（非近似），数据量少时（<10万）毫秒级返回
        # 缺点: 数据量大时（>100万）会慢，此时应换 IndexIVFFlat 等近似索引
        self.index = faiss.IndexFlatIP(self.dimension)
        self.index.add(vectors)  # 把 N 个 D 维向量加入索引

        # ---- 建立 ID 映射 ----
        # enumerate(): 给每个 chunk_id 一个递增序号
        self.id_map = {i: cid for i, cid in enumerate(chunk_ids)}

        print(f"[vector_index] 索引构建完成: {len(chunk_ids)} 个向量, {self.dimension} 维")

    def search(self, query_embedding: list[float], top_k: int = 5) -> list[tuple[str, float]]:
        """
        搜索最相似的 top_k 个 chunk。

        参数:
          query_embedding: 用户查询的向量，如 [0.01, -0.02, ...]
          top_k: 返回前 k 个结果

        返回:
          [(chunk_id, score), ...]，按相似度降序
          例如 [("dali_guide.md_abc", 0.95), ("dali_guide.md_def", 0.82)]
        """
        if self.index is None:
            print("[vector_index] 索引未构建")
            return []

        # 转成 (1, D) 的二维矩阵（FAISS 要求 batch 输入）
        query_vec = np.array([query_embedding], dtype=np.float32)

        # 归一化 query 向量（索引中的向量已归一化）
        norm = np.linalg.norm(query_vec, axis=1, keepdims=True)
        norm = np.where(norm == 0, 1.0, norm)
        query_vec = query_vec / norm

        # index.search(query, k) 返回 (distances, indices)
        # distances: (1, k) 相似度矩阵
        # indices: (1, k) 整数序号矩阵
        distances, indices = self.index.search(query_vec, top_k)

        results: list[tuple[str, float]] = []
        # 把 FAISS 返回的序号转回 chunk_id
        for dist, idx in zip(distances[0], indices[0]):
            if idx == -1:
                # FAISS 返回 -1 表示无效结果（不够 k 个）
                continue
            if idx in self.id_map:
                # float(dist): numpy float32 → Python float
                results.append((self.id_map[idx], float(dist)))

        return results

    def save(self) -> None:
        """
        把 FAISS 索引和 ID 映射序列化到磁盘。

        下次启动时直接 load()，不需要重新调 Embedding API。
        FAISS 索引用 pickle 序列化，ID 映射用 JSON 保存。
        """
        if self.index is None:
            return

        # pickle.dump() 把 Python 对象序列化成二进制文件
        # faiss.serialize_index() 或直接用 pickle 都可以
        with open(FAISS_INDEX_PATH, "wb") as f:
            pickle.dump(self.index, f)

        # json.dump() 把 dict 序列化成 JSON 文件
        with open(FAISS_ID_MAP_PATH, "w", encoding="utf-8") as f:
            json.dump(self.id_map, f, ensure_ascii=False)

        print(f"[vector_index] 索引已保存到 {FAISS_INDEX_PATH}")

    def load(self) -> bool:
        """
        从磁盘加载 FAISS 索引和 ID 映射。

        返回: True=加载成功, False=文件不存在
        """
        if not FAISS_INDEX_PATH.exists() or not FAISS_ID_MAP_PATH.exists():
            print("[vector_index] 索引文件不存在，需要重新 ingest")
            return False

        # 反序列化 FAISS 索引
        with open(FAISS_INDEX_PATH, "rb") as f:
            self.index = pickle.load(f)

        # 反序列化 ID 映射
        with open(FAISS_ID_MAP_PATH, "r", encoding="utf-8") as f:
            # json.load() 返回的 key 是字符串 "0", "1"... → 转回 int
            raw_map = json.load(f)
            self.id_map = {int(k): v for k, v in raw_map.items()}

        # 从索引中获取向量维度
        self.dimension = self.index.d

        print(f"[vector_index] 索引已加载: {len(self.id_map)} 个向量")
        return True

    @property
    def is_ready(self) -> bool:
        """索引是否可用。"""
        return self.index is not None and len(self.id_map) > 0
```

## 第十二步：`rag/retriever.py` 检索 + 重排序

**设计思路**：检索流水线——(1) Embedding 把 query 转成向量 → (2) FAISS 搜索 top_k×2 候选 → (3) 重排序精排 → (4) 回 MySQL 取完整文本 → (5) 返回格式化结果。重排序优先用规则打分（零成本、毫秒级），因为它不依赖额外 API。

```python
# ============================================================
# rag/retriever.py — 检索 + 重排序
# ============================================================

import logging

from database.chunk_repo import ChunkRepository
from rag.embedding import EmbeddingService
from rag.vector_index import VectorIndex


logger = logging.getLogger(__name__)


# ============================================================
# 规则级重排序（零成本、毫秒级）
# ============================================================

def _rule_based_score(
    query: str,
    chunk: dict[str, str],
    destination: str | None = None
) -> int:
    """
    根据 query 关键词对 chunk 打分——轻量级重排序。

    打分原则（这些规则来自对旅游攻略内容的领域理解）:
      - 标题命中 query 词： +3 分（标题匹配最相关）
      - 正文命中 query 词： +1 分
      - chunk 标题是 "文档开头"： -8 分（无标题内容通常是泛化噪音）
      - 标题含 "行程" 且不含 "行程参考"： +4 分（具体行程最有用）
      - 标题含 "行程参考"： -4 分（模板化参考内容太全面，反而稀释信息）
      - 标题含 "目的地简介"： -2 分（太泛化）
      - 目的地不匹配： -5 分（用户搜"大理"，不应该返回成都的内容）
    """
    title = chunk.get("title", "")
    text_content = chunk.get("text", "")
    source = chunk.get("source", "")

    # 把 query 按空格拆成关键词
    keywords = query.strip().split()

    score = 0
    for kw in keywords:
        if kw in title:
            score += 3     # 标题命中权重最高
        if kw in text_content:
            score += 1     # 正文命中也加分

    # ---- 降权规则 ----
    if title == "文档开头":
        score -= 8

    # ---- 加分规则 ----
    if "行程" in title and "行程参考" not in title:
        score += 4

    if "行程参考" in title:
        score -= 4

    if "目的地简介" in title:
        score -= 2

    # 目的地不匹配：降权
    if destination:
        chunk_full_text = f"{source} {title} {text_content}".lower()
        if destination.lower() not in chunk_full_text:
            score -= 5

    return score


# ============================================================
# 检索主入口
# ============================================================

class Retriever:
    """
    检索器类。

    职责:
      1. Embedding 把 query 转成向量
      2. FAISS 粗排召回候选 (top_k × 2 条)
      3. 规则打分精排（后续可扩展 LLM Rerank）
      4. 回 MySQL 取完整文本
      5. 返回格式化结果

    设计模式: 门面模式（Facade），对外只暴露出 retrieve() 一个方法。
    """

    def __init__(
        self,
        vector_index: VectorIndex,
        embedding_service: EmbeddingService,
        chunk_repo: ChunkRepository
    ):
        """
        依赖注入：所有依赖从外部传入，而不是内部创建。

        为什么这样做：
          - 方便测试：可以传入 mock 对象
          - 松耦合：换 Embedding 实现只需改外部，不动此类
          - 清晰：一眼看出这个类依赖了哪些模块
        """
        self.vector_index = vector_index
        self.embedding_service = embedding_service
        self.chunk_repo = chunk_repo

    def retrieve(
        self,
        query: str,
        top_k: int = 5,
        destination: str | None = None
    ) -> list[str]:
        """
        检索主入口。

        参数:
          query: 用户查询（已改写），如 "大理 洱海 骑行 自然风光 攻略"
          top_k: 最终返回几条
          destination: 目的地（用于规则打分）

        返回:
          格式化后的文本列表，如：
          [
            "[来源: dali_guide.md | 标题: 洱海生态廊道]\n洱海西岸的慢行系统...",
            ...
          ]
        """
        # Step 1: Query → 向量
        query_vec = self.embedding_service.embed_single(query)
        if query_vec is None:
            print("[retriever] query embedding 失败")
            return []

        # Step 2: FAISS 粗排（召回的候选数是实际需要的 2 倍，
        #          给后续精排留余量，避免漏掉）
        candidate_k = max(top_k * 2, 6)
        search_results = self.vector_index.search(query_vec, candidate_k)

        if not search_results:
            print("[retriever] FAISS 检索无结果")
            return []

        # Step 3: 回 MySQL 取完整文本
        chunk_ids = [cid for cid, _ in search_results]
        chunks = self.chunk_repo.get_chunks_by_ids(chunk_ids)

        # 构建 id → chunk 映射
        chunk_map = {c["id"]: c for c in chunks}

        # 按 FAISS 返回的顺序重建列表
        matched_chunks: list[dict[str, str]] = []
        for cid, faiss_score in search_results:
            if cid in chunk_map:
                enriched = dict(chunk_map[cid])
                enriched["faiss_score"] = round(faiss_score, 4)
                matched_chunks.append(enriched)

        # Step 4: 规则打分精排
        print("[retriever] 使用规则打分重排序")
        scored = []
        for idx, chunk in enumerate(matched_chunks):
            s = _rule_based_score(query, chunk, destination=destination)
            # -idx 作为次要排序键：同分时保留 FAISS 原始顺序
            scored.append((s, -idx, chunk))

        # 按分数降序排序
        scored.sort(key=lambda x: (x[0], x[1]), reverse=True)

        # 取 top_k
        selected = [chunk for _, _, chunk in scored[:top_k]]

        # Step 5: 格式化成可读文本
        results: list[str] = []
        for chunk in selected:
            formatted = (
                f"[来源: {chunk['source']} | 标题: {chunk['title']}]\n"
                f"{chunk['text']}"
            )
            results.append(formatted)

        print(f"[retriever] 检索完成，返回 {len(results)} 条")
        return results
```

## 第十三步：`agents/rag_tool.py` Query Rewrite + 上下文组装

**设计思路**：用户输入自然语言（"大理慢节奏看日落"），不能直接当搜索关键词。需要用 LLM 把自然语言改写成适合向量检索的关键词组合（Query Rewrite）。然后调 Retriever 拉回上下文。

```python
# ============================================================
# agents/rag_tool.py — Query Rewrite + RAG 上下文
# ============================================================

import logging

from config import (
    LLM_API_KEY,
    LLM_BASE_URL,
    LLM_MAX_RETRIES,
    LLM_MODEL,
    LLM_TIMEOUT_SECONDS,
)
from rag.retriever import Retriever


logger = logging.getLogger(__name__)


def _llm_rewrite_query(
    destination: str,
    preferences: list[str] | None = None,
    pace: str | None = None,
    special_notes: str | None = None,
) -> str | None:
    """
    用 LLM 把自然语言需求改写成检索关键词。

    为什么需要这一步：
      用户输入 "大理慢节奏看日落拍照之旅" 是自然语言，直接向量检索效果差。
      因为攻略文本里不会写"慢节奏之旅"，而是写"洱海生态廊道适合骑行，日落时分非常出片"。
      LLM 把自然语言 → 信息密集的关键词（如 "大理 洱海 双廊 日落 拍照 轻松 攻略"），
      检索命中率大幅提升。

    参数:
      destination: 目的地
      preferences: 偏好列表
      pace: 节奏
      special_notes: 额外备注

    返回:
      改写后的查询字符串，或 None（失败时）
    """
    if not LLM_API_KEY:
        return None

    # 延迟导入：只在需要时导入 LangChain，加快模块加载
    try:
        from langchain_openai import ChatOpenAI
    except ImportError:
        logger.warning("langchain_openai 未安装，无法使用 LLM Query Rewrite")
        return None

    # 创建 LLM 实例
    # temperature=0.2: 低温度让输出更确定、一致（不需要创造性）
    llm = ChatOpenAI(
        model=LLM_MODEL,
        temperature=0.2,
        api_key=LLM_API_KEY,
        base_url=LLM_BASE_URL or None,
        timeout=LLM_TIMEOUT_SECONDS,
        max_retries=LLM_MAX_RETRIES,
    )

    # system prompt: 设定 LLM 的角色和行为
    system_prompt = (
        "你是一个 RAG 检索 query 改写专家。"
        "你的任务是把用户的旅行需求改写成适合向量检索的关键词组合。"
        "规则："
        "1. 只输出关键词，用空格分隔"
        "2. 不要输出解释、标点或任何多余文字"
        "3. 关键词要具体，优先包含：景点名称、活动类型、场景特征"
        "4. 必须包含目的地城市名"
    )

    # human prompt: 具体任务输入
    parts = [f"目的地：{destination}"]
    if preferences:
        parts.append(f"偏好：{'、'.join(preferences)}")
    if pace:
        parts.append(f"节奏：{pace}")
    if special_notes:
        parts.append(f"备注：{special_notes}")

    try:
        # llm.invoke(): 同步调用 LLM
        # 参数格式是 LangChain 的消息列表: [("role", "content"), ...]
        response = llm.invoke([
            ("system", system_prompt),
            ("human", "\n".join(parts)),
        ])

        # 提取响应文本
        raw = response.content if hasattr(response, "content") else str(response)
        if isinstance(raw, list):
            raw = "".join(str(x) for x in raw)

        query = raw.strip()
        if query:
            logger.info(f"[rewrite] 输入: {parts} → 输出: {query}")
            return query
    except Exception as exc:
        logger.warning(f"[rewrite] LLM 调用失败: {exc}")

    return None


def _rule_rewrite_query(
    destination: str,
    preferences: list[str] | None = None,
    pace: str | None = None,
    special_notes: str | None = None,
) -> str:
    """
    规则级 Query Rewrite（LLM 不可用时的 fallback）。

    用预定义的映射表从备注中提取关键词，不需要调 LLM，零成本。
    """
    keywords: list[str] = [destination]

    if preferences:
        for p in preferences:
            if p not in keywords:
                keywords.append(p)

    if pace:
        keywords.append(pace)

    # 备注关键词 → 检索关键词的映射
    # 例如用户备注 "想看日落" → 检索词加上 "日落 傍晚 双廊 洱海"
    note_keyword_map = {
        "日落": ["日落", "傍晚", "双廊", "洱海"],
        "日出": ["日出", "清晨", "才村"],
        "拍照": ["拍照", "摄影", "出片"],
        "美食": ["美食", "小吃", "餐饮"],
        "轻松": ["轻松", "慢节奏", "休闲"],
        "骑行": ["骑行", "洱海生态廊道"],
        "古镇": ["古镇", "古城"],
        "熊猫": ["大熊猫", "熊猫基地"],
        "潜水": ["潜水", "水上项目"],
    }

    if special_notes:
        for trigger, values in note_keyword_map.items():
            if trigger in special_notes:
                for v in values:
                    if v not in keywords:
                        keywords.append(v)

    # 加上通用检索增强词（提高召回率）
    for term in ["景点", "行程", "攻略", "推荐"]:
        if term not in keywords:
            keywords.append(term)

    return " ".join(keywords)


def get_destination_guide_context(
    destination: str,
    preferences: list[str] | None = None,
    pace: str | None = None,
    special_notes: str | None = None,
    top_k: int = 5,
    retriever: Retriever | None = None,
) -> list[str]:
    """
    对外接口：根据旅行需求返回本地攻略上下文。

    流程:
      1. Query Rewrite（LLM 优先、规则 fallback）
      2. 调 Retriever 检索
      3. 返回格式化文本列表

    参数:
      destination: 目的地
      preferences: 偏好
      pace: 节奏
      special_notes: 备注
      top_k: 返回几条
      retriever: Retriever 实例（依赖注入）

    返回:
      格式化文本列表，每条是 "[来源: xxx | 标题: xxx]\n正文..."
    """
    # Step 1: Query Rewrite
    query = _llm_rewrite_query(
        destination=destination,
        preferences=preferences,
        pace=pace,
        special_notes=special_notes,
    )
    if not query:
        query = _rule_rewrite_query(
            destination=destination,
            preferences=preferences,
            pace=pace,
            special_notes=special_notes,
        )

    print(f"[rag_tool] 检索 query: {query}")

    if retriever is None:
        print("[rag_tool] retriever 未初始化")
        return []

    # Step 2: 检索
    contexts = retriever.retrieve(
        query=query,
        top_k=top_k,
        destination=destination,
    )

    print(f"[rag_tool] 检索到 {len(contexts)} 条上下文")
    return contexts

## 第十四步：`agents/trip_planner_agent.py` Agent 行程生成

**设计思路**：这是 Agent 的核心——把 RAG 上下文 + 用户需求一起给 LLM，让它生成结构化 JSON。用 **Pydantic 做输出校验**，确保 LLM 返回的 JSON 符合预期结构。

```python
# ============================================================
# agents/trip_planner_agent.py — LLM 行程生成 Agent
# ============================================================

import json

# Pydantic 模型：校验 LLM 输出的 JSON 结构
from pydantic import BaseModel, Field

from config import (
    LLM_API_KEY,
    LLM_BASE_URL,
    LLM_MAX_RETRIES,
    LLM_MODEL,
    LLM_TIMEOUT_SECONDS,
)
from models.schemas import TripRequest


# ============================================================
# LLM 输出的中间数据模型
# 为什么不用 schemas.py 里的最终模型：
#   LLM 只负责产出核心信息（每天的主题、景点、餐饮），
#   预算、酒店、交通等细节由 trip_service.py 补充。
#   这样 LLM 的输出结构简单，不易出错。
# ============================================================

class PlannerDayDraft(BaseModel):
    """LLM 返回的单日草稿——每天的精华信息。"""
    day_index: int = Field(..., ge=1)          # 第几天，≥1
    theme: str                                   # 当天主题，如 "洱海慢骑行"
    spot_name: str                               # 主景点名
    spot_description: str                        # 推荐理由
    meal_name: str                               # 餐饮建议

class PlannerDraft(BaseModel):
    """LLM 返回的完整行程草稿。"""
    summary: str                                 # 概述
    tips: list[str] = Field(default_factory=list)  # 旅行提示
    days: list[PlannerDayDraft] = Field(default_factory=list)


# ============================================================
# JSON 提取工具
# LLM 有时候会用 ```json ... ``` 包裹输出，需要清理
# ============================================================

def _extract_json_from_text(raw_text: str) -> str | None:
    """
    从 LLM 的原始输出中提取 JSON 对象。

    处理三种情况:
      1. 正常 JSON: {"summary": "..."}
      2. Markdown 代码块: ```json\n{...}\n```
      3. 带解释文字: 前面有一些文字 {...} 后面有一些文字
    """
    text = raw_text.strip()

    # 去掉 Markdown 代码块包裹
    if text.startswith("```"):
        lines = text.splitlines()
        # 去掉第一行（```json 或 ```）
        if lines and lines[0].startswith("```"):
            lines = lines[1:]
        # 去掉最后一行（```）
        if lines and lines[-1].strip() == "```":
            lines = lines[:-1]
        text = "\n".join(lines).strip()
        # 如果第一行是 "json"，也去掉
        if text.lower().startswith("json"):
            text = text[4:].strip()

    # 找到最外层 { 和 }
    start = text.find("{")
    end = text.rfind("}")
    if start == -1 or end == -1 or end <= start:
        return None
    return text[start : end + 1]


# ============================================================
# Chat LLM 工厂函数
# ============================================================

def _build_chat_llm():

    创建 LangChain ChatOpenAI 实例。

    为什么抽成单独函数:
      多个地方需要创建 LLM（生成草稿、Query Rewrite），
      统一工厂函数避免散落重复代码。
    """
    if not LLM_API_KEY:
        return None

    try:
        from langchain_openai import ChatOpenAI
    except ImportError:
        return None

    return ChatOpenAI(
        model=LLM_MODEL,                 # deepseek-v4-flash
        temperature=0.3,                 # 0.3: 低温度保证输出稳定、可控
        api_key=LLM_API_KEY,
        base_url=LLM_BASE_URL or None,   # https://api.deepseek.com/v1
        timeout=LLM_TIMEOUT_SECONDS,
        max_retries=LLM_MAX_RETRIES,
    )


# ============================================================
# 生成行程草稿
# ============================================================

def generate_planner_draft(
    request: TripRequest,
    rag_contexts: list[str],
    day_count: int,
) -> tuple[PlannerDraft | None, dict[str, int]]:
    """
    调用 LLM 生成结构化行程草稿。

    参数:
      request: 用户旅行需求
      rag_contexts: RAG 检索到的攻略上下文（已格式化的文本列表）
      day_count: 总天数

    返回:
      tuple[PlannerDraft | None, dict]:
        - PlannerDraft: 成功时为草稿对象，失败为 None
        - dict: {"prompt_tokens": N, "completion_tokens": N}
    """
    empty_usage = {"prompt_tokens": 0, "completion_tokens": 0}

    llm = _build_chat_llm()
    if llm is None:
        print("[agent] 无法创建 LLM 实例，请检查 LLM_API_KEY")
        return None, empty_usage

    # 把多条上下文拼成一段
    guide_context = "\n\n".join(rag_contexts) if rag_contexts else "暂无本地攻略上下文。"

    # ---- System Prompt ----
    # 设定 LLM 的角色、行为规则和输出格式
    system_prompt = (
        "你是一名旅行规划助手。"
        "请用中文生成简洁的结构化旅行草稿。"
        "你需要严格根据用户给出的目的地、预算、节奏和本地攻略来规划。"
        "你必须只输出一个 JSON 对象，不要输出 Markdown、解释文字或代码块。"
        "如果用户有特殊备注（如想看日落、不想太早起），必须体现在具体某一天的安排里。"
        "每天的安排要符合给定的节奏，避免太满、太赶。"
        "餐饮建议优先使用攻略里出现过的特色餐饮。"
    )

    # ---- Human Prompt ----
    # 把所有用户信息 + RAG 上下文 + 输出要求注入
    human_prompt = f"""
目的地：{request.destination}
出发日期：{request.start_date.isoformat()}
结束日期：{request.end_date.isoformat()}
天数：{day_count}
人数：{request.travelers}
预算：{request.budget}
偏好：{'、'.join(request.preferences) if request.preferences else '无特别偏好'}
节奏：{request.pace or '适中'}
额外备注：{request.special_notes or '无'}

本地攻略上下文：
{guide_context}

要求：
1. 输出一个整体 summary（概述，1-2 句话）。
2. 输出 {day_count} 天的 daily draft。
3. 每天只给一个主要景点、一个餐饮建议和一条当天备注。
4. tips 保持简洁（2-3 条）。
5. day_index 必须从 1 到 {day_count}。
6. 额外备注中的要求必须在 days 中体现，不要只写在 tips 里。
7. 只返回 JSON 对象，不要使用 ```json 代码块。

JSON 结构示例：
{{
  "summary": "整体概述",
  "tips": ["提示1", "提示2"],
  "days": [
    {{
      "day_index": 1,
      "theme": "当天主题",
      "spot_name": "主要景点",
      "spot_description": "景点推荐理由",
      "meal_name": "餐饮名称",
      "meal_notes": "餐饮说明",
      "daily_note": "当天备注"
    }}
  ]
}}
"""

    print(f"[agent] 正在调用 {LLM_MODEL} 生成行程...")

    try:
        # llm.invoke(): 发送请求，等待完整响应后返回
        response = llm.invoke([
            ("system", system_prompt),
            ("human", human_prompt),
        ])
    except Exception as exc:
        print(f"[agent] LLM 调用失败: {exc}")
        return None, empty_usage

    # ---- 提取 token 用量 ----
    # LangChain 的 AIMessage 对象把 token 用量存在 response_metadata 里
    metadata = getattr(response, "response_metadata", None) or {}
    token_info = metadata.get("token_usage", {})
    token_usage = {
        "prompt_tokens": token_info.get("prompt_tokens", 0),
        "completion_tokens": token_info.get("completion_tokens", 0),
    }

    # ---- 提取 JSON ----
    # AIMessage.content 是 LLM 返回的文本
    raw_text = response.content if hasattr(response, "content") else str(response)
    # 有些 API 把 content 返回成 list（OpenAI 流式），这里统一处理
    if isinstance(raw_text, list):
        raw_text = "".join(str(x) for x in raw_text)

    json_text = _extract_json_from_text(str(raw_text))
    if json_text is None:
        print(f"[agent] 无法从 LLM 返回中提取 JSON")
        print(f"[agent] 原始返回预览: {str(raw_text)[:300]}")
        return None, token_usage

    # ---- Pydantic 校验 ----
    try:
        # json.loads(): JSON 字符串 → Python dict
        # PlannerDraft.model_validate(): dict → Pydantic 对象，自动类型校验
        result = PlannerDraft.model_validate(json.loads(json_text))
    except Exception as exc:
        print(f"[agent] JSON 解析或校验失败: {exc}")
        print(f"[agent] JSON 原文: {json_text[:300]}")
        return None, token_usage

    # ---- 天数校验 ----
    if len(result.days) != day_count:
        print(f"[agent] 天数不匹配: 期望 {day_count} 天，LLM 返回 {len(result.days)} 天")
        return None, token_usage

    print(f"[agent] 行程生成成功！token: prompt={token_usage['prompt_tokens']}, completion={token_usage['completion_tokens']}")
    return result, token_usage

def _estimate_ticket_cost(spot_name: str, description: str | None = None) -> float:
    """
    根据景点名称中的关键词粗略估算门票价格。

    为什么不用真实 API 查门票：
      这是 MVP，先用规则粗略估算。后续可以接地图 API 拿真实数据。
    """
    text = f"{spot_name} {description or ''}"

    # any(kw in text): 只要 text 中包含任意一个关键词就命中
    if any(kw in text for kw in ("古城", "古镇", "公园", "廊道", "村", "湿地", "街区")):
        return 0.0         # 免费景点
    if any(kw in text for kw in ("寺", "塔", "博物馆", "遗址")):
        return 75.0        # 文化类
    if any(kw in text for kw in ("索道", "缆车", "游船", "演出", "水上", "乐园")):
        return 150.0       # 体验类
    return 50.0            # 默认


def _build_fallback_spots(destination: str, rag_contexts: list[str], count: int) -> list[str]:
    """
    LLM 故障时使用的硬编码回退景点表。

    设计考虑：
      LLM API 可能超时/限流/返回格式错误——此时用这个表兜底，
      保证用户在任何情况下都能拿到一份（虽然不如 AI 生成的那么个性化）的计划。
    """
    city_spot_map = {
        "大理": ["大理古城", "洱海生态廊道", "喜洲古镇", "双廊古镇", "崇圣寺三塔"],
        "成都": ["大熊猫繁育基地", "宽窄巷子", "锦里", "都江堰", "青城山"],
        "三亚": ["亚龙湾", "蜈支洲岛", "天涯海角", "南山文化旅游区", "亚特兰蒂斯水世界"],
        "厦门": ["鼓浪屿", "环岛路", "曾厝垵", "南普陀寺", "沙坡尾"],
        "西安": ["兵马俑", "大雁塔", "回民街", "城墙", "大唐不夜城"],
    }

    candidates = city_spot_map.get(
        destination,
        [f"{destination}推荐景点 {i+1}" for i in range(count)]
    )

    # 确保有 count 个：不够就补 "第N天" 占位
    while len(candidates) < count:
        candidates.append(f"{destination} 第{len(candidates) + 1}天")
    return candidates[:count]

# 编排主函数
# ============================================================

class TripService:
    """
    行程编排服务。

    设计模式: Service Layer（服务层）
    - 不直接操作数据库
    - 不直接调用 LLM
    - 只做流程编排：RAG → Agent → 组装 → 返回

    依赖注入：
      所有底层组件通过构造函数传入，方便测试和替换。
    """

    def __init__(
        self,
        retriever: Retriever,
        chunk_repo: ChunkRepository,
    ):
        self.retriever = retriever
        self.chunk_repo = chunk_repo

    def generate_itinerary(self, request: TripRequest) -> Itinerary:
        """
        根据用户请求生成完整行程。

        完整流程:
          Step 1: 计算天数
          Step 2: RAG 检索攻略上下文
          Step 3: Agent 调用 LLM 生成草稿
          Step 4: 组装 DayPlan（LLM 草稿 + 规则兜底）
          Step 5: 计算预算
          Step 6: 生成 summary & tips
          Step 7: 组装最终 Itinerary 对象
        """
        # ---- Step 1: 计算天数 ----
        # (end_date - start_date).days: timedelta 的天数属性
        # +1: 例如 6/1 到 6/3 是 3 天（1号、2号、3号）
        day_count = (request.end_date - request.start_date).days + 1
        day_count = max(day_count, 1)   # 至少 1 天

        # ---- Step 2: RAG 检索 ----
        rag_contexts = get_destination_guide_context(
            destination=request.destination,
            preferences=request.preferences,
            pace=request.pace,
            special_notes=request.special_notes,
            top_k=5,
            retriever=self.retriever,
        )

        # ---- Step 3: Agent 生成草稿 ----
        llm_draft, planner_usage = generate_planner_draft(
            request, rag_contexts, day_count
        )

        token_usage = TokenUsage(
            planner_prompt=planner_usage.get("prompt_tokens", 0),
            planner_completion=planner_usage.get("completion_tokens", 0),
        )

        # ---- Step 4: 组装每天的 DayPlan ----
        fallback_spots = _build_fallback_spots(
            request.destination, rag_contexts, day_count
        )

        days: list[DayPlan] = []
        ticket_total = 0.0

        for i in range(day_count):
            day_number = i + 1
            # timedelta(days=i): 从 start_date 往后推 i 天
            current_date = request.start_date + timedelta(days=i)

            # 从 LLM 草稿中找对应的那一天
            llm_day = None
            if llm_draft is not None:
                # next(..., None): 取第一个匹配的，没有返回 None
                llm_day = next(
                    (d for d in llm_draft.days if d.day_index == day_number),
                    None,
                )

            # 优先用 LLM 生成的数据，失败则用规则兜底
            spot_name = llm_day.spot_name if llm_day else fallback_spots[i]
            theme = llm_day.theme if llm_day else f"第{day_number}天：{request.destination}轻松游"
            spot_desc = (
                llm_day.spot_description
                if llm_day
                else "根据本地攻略和旅行偏好安排，适合半天慢慢游览。"
            )
            meal_name = llm_day.meal_name if llm_day else f"{request.destination}特色餐饮"
            meal_note = llm_day.meal_notes if llm_day else "根据攻略推荐。"
            daily_note = llm_day.daily_note if llm_day else "今天以轻松游览为主，灵活调整。"
            ticket_cost = _estimate_ticket_cost(spot_name, spot_desc)
            ticket_total += ticket_cost

            day_plan = DayPlan(
                day_index=day_number,
                date=current_date,
                theme=theme,
                spots=[
                    SpotItem(
                        name=spot_name,
                        description=spot_desc,
                        estimated_cost=ticket_cost,
                        location=request.destination,
                    )
                ],
                meals=[
                    MealItem(
                        name=meal_name,
                        meal_type="午餐",
                        estimated_cost=80.0,
                        notes=meal_note,
                    )
                ],
                hotel=HotelItem(
                    name=f"{request.destination}舒适型住宿",
                    level="舒适型",
                    estimated_cost=300.0,
                    location=f"{request.destination}市区",
                ),
                transport=[
                    TransportItem(
                        mode="打车",
                        from_place="出发点",
                        to_place=spot_name,
                        estimated_cost=50.0,
                        duration="30分钟",
                    )
                ],
                notes=[f"节奏：{request.pace or '适中'}", daily_note],
            )
            days.append(day_plan)

        # ---- Step 5: 预算 ----
        total_budget = request.budget
        hotel_total = day_count * 300.0
        meal_total = day_count * 80.0
        transport_total = day_count * 50.0
        other = max(0, total_budget - hotel_total - meal_total - transport_total - ticket_total)

        # ---- Step 6: summary & tips ----
        if llm_draft:
            summary = llm_draft.summary
            tips = llm_draft.tips
        else:
            prefs = "、".join(request.preferences) if request.preferences else "常规旅行"
            summary = f"{request.destination} {day_count} 日{prefs}之旅。"
            tips = [
                f"建议根据{request.destination}当天天气准备合适衣物。",
                "热门景点建议错峰出发，预留拍照和用餐缓冲时间。",
            ]

        # ---- Step 7: 组装最终 Itinerary ----
        itinerary = Itinerary(
            trip_id=f"trip_{request.destination}_{request.start_date.isoformat()}",
            destination=request.destination,
            summary=summary,
            days=days,
            estimated_budget=total_budget,
            budget_breakdown=BudgetBreakdown(
                transport=round(transport_total, 2),
                hotel=round(hotel_total, 2),
                meals=round(meal_total, 2),
                tickets=round(ticket_total, 2),
                other=round(other, 2),
                total=round(total_budget, 2),
            ),
            tips=tips,
            source_notes=rag_contexts[:3],
            token_usage=token_usage,
        )

        print(f"[service] 行程生成完毕: {request.destination} {day_count} 天")
        return itinerary
```

## 设计模式总结

| 模式 | 应用位置 | 作用 |
|------|---------|------|
| **单例** | `config.py`（模块级变量）、`database/connection.py`（engine） | 全局唯一实例，避免重复创建 |
| **DAO** | `database/chunk_repo.py` | 数据库访问封装，上层不接触 SQL |
| **门面** | `rag/retriever.py`、`services/trip_service.py` | 对外暴露简单接口，隐藏内部复杂性 |
| **策略** | `rag/embedding.py`（可替换实现） | 不同 Embedding 提供商可互换 |
| **依赖注入** | `Retriever.__init__`、`TripService.__init__` | 组件从外部注入，松耦合、易测试 |
| **模型校验** | `models/schemas.py`（Pydantic BaseModel） | 数据进入系统时自动校验，错误及早发现 |

## 常见问题

**Q: 我只有 DeepSeek API Key，没有单独的 Embedding API，怎么办？**

A: SiliconFlow 注册即送免费额度，足够开发使用。注册地址 https://siliconflow.cn ，在"API 密钥"页面创建 Key 填入 `.env` 的 `EMBEDDING_API_KEY`。

**Q: 我已经有 OpenAI API Key，能不能用 OpenAI 的 Embedding？**

A: 可以。`.env` 中改成：
```
EMBEDDING_API_KEY=sk-your-openai-key
EMBEDDING_MODEL=text-embedding-3-small
EMBEDDING_BASE_URL=https://api.openai.com/v1
```

**Q: FAISS 索引文件存在哪里？**

A: `zhilv-yuntu/db/faiss_index.bin` 和 `faiss_id_map.json`。删除了重新 `python main.py --ingest` 即可重建。

**Q: ingest 需要多久？**

A: 取决于攻略文件数量和 Embedding API 速度。5 个攻略文件约 40-50 个 chunk，SiliconFlow 的 Embedding API 批量处理约 2-5 秒。
